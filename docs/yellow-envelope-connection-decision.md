# 노란봉투법.com ↔ 지원금 사이트 연결 방식 결정안

## 목적

- `employment-subsidy-guide`와 `노란봉투법.com`을 어떤 수준으로 연결할지 결정한다.
- 구현 착수 전에 연결 전략을 닫아서 재작업을 줄인다.
- 판단 기준은 `신뢰`, `전환`, `운영 난이도`, `SEO`, `확장성`이다.

## 현재 상황

- 지원금 사이트는 `고용장려금 탐색 -> 자격 자가진단 -> 후속 상담 전환` 흐름이 중심이다.
- 노란봉투법.com은 별도 제품/도메인으로 운영 중이거나 운영 예정인 연결 대상이다.
- 현재 병목은 구현이 아니라 **연결 수준을 어디까지 할지**에 대한 결정이다.

## 평가 기준

1. 빠르게 붙일 수 있는가
2. 사용자 경험이 분절되지 않는가
3. SEO/검색 유입에 손해가 없는가
4. 운영 복잡도가 감당 가능한가
5. 나중에 더 강한 통합으로 확장하기 쉬운가

## 옵션 비교

### 옵션 A. 단순 링크 연결

설명:
- 지원금 사이트 내 적절한 지점에 `노란봉투법.com`으로 가는 링크/CTA를 둔다.
- 예: 홈, 상세, 결과 페이지, 상담 전환 직전 또는 완료 후 안내.

장점:
- 가장 빠르다
- 구현 난이도와 운영 리스크가 가장 낮다
- SEO/성능에 미치는 부작용이 적다
- 우선 실서비스에서 반응을 보기 좋다

단점:
- 사용 경험이 분리된다
- 외부 이동 시 이탈 가능성이 크다
- 제품 통합감이 약하다

적합한 경우:
- 이번 주 안에 붙여야 할 때
- 어떤 흐름이 맞는지 아직 실험 단계일 때

### 옵션 B. API 기반 데이터 연동

설명:
- 두 서비스가 API/DB를 통해 일부 데이터 또는 상태를 주고받는다.
- 추천, 개인화, 공통 검색, 상담 맥락 연결 같은 확장이 가능하다.

장점:
- 경험 일관성이 좋아진다
- 장기적으로 제품 간 시너지가 가장 크다
- 정교한 추천/분기 설계가 가능하다

단점:
- 설계와 구현 비용이 크다
- 인증, 스키마, 장애 대응 등 운영 복잡도가 올라간다
- 지금 당장 결정해야 할 항목이 많다

적합한 경우:
- 장기 메인 전략으로 두 제품을 묶을 때
- 공통 사용자 흐름과 데이터 모델이 이미 정리됐을 때

### 옵션 C. 임베드/iframe형 연결

설명:
- 한쪽 서비스를 다른 쪽 안에 iframe 또는 임베드 블록처럼 보여준다.

장점:
- 겉보기 통합은 빠르다
- 별도 백엔드 통합 없이도 시도 가능하다

단점:
- UX가 어색해지기 쉽다
- SEO에 불리하다
- 반응형, 상호작용, 스타일 일관성, 분석 추적이 꼬이기 쉽다
- 장기 유지보수성이 낮다

적합한 경우:
- 임시 실험용
- 단기 데모 목적

### 옵션 D. 콘텐츠/라우팅 레벨 통합

설명:
- 두 제품을 하나의 제품군처럼 재구성한다.
- 라우팅, 콘텐츠 흐름, CTA 구조까지 통합 설계한다.

장점:
- 사용자 입장에서는 가장 자연스럽다
- 브랜딩/탐색/전환 흐름이 가장 일관된다
- 장기적으로 제품 완성도가 높다

단점:
- 설계와 개발량이 가장 크다
- 지금 바로 착수하면 범위가 커질 위험이 있다
- 의사결정 미완료 상태에서 들어가면 재작업이 크다

적합한 경우:
- 장기적으로 한 제품처럼 운영할 계획이 명확할 때

## PM 권고안

### 권고: `옵션 A`로 먼저 붙이고, 구조는 `옵션 D`를 염두에 두고 설계

권고 이유:
- 현재는 연결 방식 자체가 병목이다.
- 지금 필요한 건 **빠른 구현 + 낮은 재작업 위험**이다.
- API 통합(B)이나 구조 통합(D)은 장기적으로 좋지만, 지금 바로 들어가면 범위가 커진다.
- iframe(C)은 빠르지만 장기적으로 UX/SEO/유지보수 손해가 크다.

따라서 현재 최적안은:
1. **단순 링크 연결(A)** 로 우선 출시 가능한 형태를 만든다.
2. 다만 CTA 위치, 문구, 경로 설계는 나중에 **콘텐츠/라우팅 통합(D)** 으로 확장 가능하게 잡는다.

## 추천 구현 방식 (1차)

### 연결 위치 후보

1. 홈 화면
- 고용장려금 외 다른 노동 이슈 탐색 진입점으로 노출

2. 자격검토 결과 화면
- 사용자가 추가 노동 이슈 정보를 찾을 수 있는 보조 CTA로 연결

3. 상담 전환 박스 근처
- “관련 노동 분쟁/판례 정보도 함께 확인” 성격의 보조 링크

4. 상세 페이지 하단
- 관련 사이트/후속 탐색 리소스로 연결

### 카피 방향

- 과장 없이, 보조 리소스처럼 보여야 함
- 예:
  - `노동 이슈 판례와 해설도 함께 보기`
  - `노란봉투법 관련 정보는 별도 사이트에서 확인할 수 있습니다`
  - `관련 노동 판례/쟁점도 이어서 확인해보세요`

### 피해야 할 것

- 메인 CTA를 빼앗는 과도한 외부 유도
- iframe 임시 통합
- 아직 정의되지 않은 API 연동 전제 문구
- 두 사이트가 이미 완전 통합된 것처럼 보이게 만드는 과장 표현

## 후속 단계

### Phase 1
- 링크 연결 구현
- 클릭 위치/유입/이탈 데이터 관찰

### Phase 2
- 어떤 페이지에서 연결이 가장 유효한지 판단
- 필요 시 공통 브랜딩/라우팅 문구 정리

### Phase 3
- 장기적으로 API 통합(B) 또는 콘텐츠/라우팅 통합(D) 검토

## 현재 결정 제안

- **지금 결정:** 옵션 A 채택
- **보류:** B/C/D는 장기 검토
- **즉시 실행 가능 작업:** 지원금 사이트 내 연결 CTA 설계 및 구현

## 실행 메모

- 공개 UX에서는 메인 전환 흐름(`탐색 -> 자가진단 -> 상담`)을 해치지 않도록 보조 연결로 취급한다.
- 이후 실사용 데이터를 본 뒤 강한 통합으로 갈지 재판단한다.
